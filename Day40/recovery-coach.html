How this was built

The system prompt

The assistant is instructed to act as a calm, evidence-informed habit & recovery coach for beginners — not a clinician. It's told to read one piece of free text describing sleep/stress/energy struggles and return exactly one small, three-part routine (morning / midday / evening), each step short enough to actually start today. It's explicitly forbidden from diagnosing conditions, prescribing supplements or medication, or giving a wall of options — the whole point is to reduce a beginner's decision fatigue to one clear next step per part of the day.

Output format

The model is instructed to reply with strict JSON only (a headline, a one-line summary, three routine blocks with a time label/title/reasoning, and a short grounding note). This lets the interface render a real timeline UI instead of a wall of chat text — matching the "structured plan" output style requested.

Edge cases handled in the prompt

  • Off-topic input (e.g. asking for code help) → the model returns a JSON object with a polite redirect instead of forcing a routine.
  • Too little information → the model still returns a gentle, general-purpose starter routine rather than blocking the user, but keeps it deliberately generic.
  • Medical red flags (chest pain, persistent insomnia, mentions of self-harm) → the model is instructed to skip routine-building and instead give a brief, caring line pointing to a doctor or appropriate support, never attempting to handle it itself.
  • Abuse/prompt injection → the model is told to ignore instructions embedded in user text that try to change its role, and to stay in character.

Why this UI

Sleep, stress, and energy all move across a day, so the interface leans on a single visual metaphor — a day's arc that resolves into a morning/midday/evening timeline — rather than a generic scorecard or chat bubbles. That keeps the tone calm and non-clinical, matching the "professional but kind" personality from the brief.

How to extend this

  • Memory: store each day's routine in localStorage-equivalent (a JS object, since artifacts can't use real localStorage) and let the model reference "yesterday's routine" for gentle continuity.
  • Multi-step flow: turn this into a 3-day check-in loop — after showing the routine, ask "how did last night go?" and feed that answer back into the next API call.
  • Tools: connect a calendar MCP tool so the model can actually schedule the wind-down block instead of just suggesting a time.
  • Personalization: add optional quick-select chips (chronotype, work schedule) before the free-text box to reduce ambiguity for the model.
